## 🎯 Filosofía del kit

ZCode tiene ~63 skills instaladas. **La mayoría son para nichos específicos** (PDFs, Excel, plugins, hooks, etc.). Este kit es **opinionated**: contiene solo las skills transversales que vas a usar en cualquier proyecto de software, desarrollo web, automatización o diseño profesional.

Además de las 24 skills, este kit incluye tres **políticas transversales críticas**:

1. **🖼️ Política de assets visuales** — Cuando un proyecto necesita imágenes, videos o animaciones que no se pueden generar en código, el sistema sabe cómo reaccionar: ofrece prompts listos, busca en internet, o delega al usuario.
2. **🎵 Política de audio (música + sonidos)** — Distingue claramente: MÚSICA (lo provee el usuario vía Suno/Udio) y SONIDOS (skill `sound-fx-bank` con catálogo desde bancos libres). Respetar accesibilidad y mute.
3. **💾 Política de backup** — Antes de cualquier cambio significativo, el sistema crea una copia de seguridad con timestamp y la mantiene en rotación.

> 💡 **Tip Pro:** Usa estas 24 skills + 3 políticas como tu kit base. Agrega el resto cuando el proyecto lo demande.

## 🖼️ Política de assets visuales — Imágenes, videos y animaciones

Cuando un proyecto necesita **imágenes, videos o animaciones** que no se pueden generar como SVG/JS en código, ZCode sigue esta política de 3 pasos:

### Paso 1 — Intentar generar en código (preferido)

> **Regla de oro:** Si el asset es decorativo, ícono, logo, botón, efecto visual, animación corta o gráfico → **generarlo en código**.

- **SVG inline** para logos, íconos, emblemas, coronas, sellos, partículas.
- **CSS keyframes / GSAP** para animaciones decorativas, transiciones, efectos de luz.
- **Canvas API** para confeti, fondos procedurales, simulaciones.
- **CSS gradients** para fondos, brillos, texturas.

Esto evita dependencias externas, problemas de licencia, lentitud de carga y resultados impredecibles.

### Paso 2 — Si NO se puede generar en código, evaluar opciones

Si el asset es **orgánico, fotográfico, ilustración compleja, video, motion graphics o audio**, evaluar:

| Opción | Cuándo usarla | Quién lo hace | Costo |
|--------|---------------|---------------|-------|
| **IA generativa** | Cuando quieres algo único y personalizado (logos conceptuales, escenas temáticas, retratos) | ZCode redacta el prompt y/o ejecuta la generación | Medio (créditos de IA) |
| **Búsqueda web** | Cuando necesitas fotos reales (paisajes, naturaleza, arquitectura, referencia) | ZCode busca en Unsplash, Pexels, Pixabay, Wikimedia | Gratis (con atribución) |
| **El usuario** | Cuando quieres control total,素材 propio (fotos familiares, branding corporativo, material con copyright) | El usuario aporta el archivo | Gratis pero requiere acción del usuario |

### Paso 3 — ZCode SIEMPRE da prompts listos

**Incluso si el usuario va a generar el asset él mismo o lo va a buscar**, ZCode debe:

1. ✅ Redactar un **prompt optimizado** listo para copiar/pegar en Midjourney, DALL-E, Stable Diffusion, Sora, Runway, Kling, etc.
2. ✅ Ofrecer **2-3 variaciones de prompt** (estilo fotorrealista, ilustración, minimalista, cinematográfico, etc.).
3. ✅ Especificar **aspect ratio, resolución y duración** según el destino del asset.
4. ✅ Indicar el **nombre del archivo** sugerido y la **carpeta destino** en el proyecto.
5. ✅ Incluir **criterios de aceptación** para que el usuario sepa cuándo el asset está bien.

### 🧪 Tests que ZCode debe correr al integrar un asset

Después de generar/recibir el asset:

1. ✅ Verificar que el archivo existe en la ruta correcta.
2. ✅ Verificar dimensiones y peso del archivo.
3. ✅ Verificar coherencia visual con la paleta del proyecto.
4. ✅ Hacer una **prueba en pantalla**: ¿se ve bien al lado de los demás elementos?
5. ✅ Documentar el asset en el `MEMORIA_PROYECTO.md` (origen, prompt usado, licencia).

### 🚫 Errores que NO se repiten con assets

1. ❌ **No asumir** que las generaciones de imagen del usuario saldrán como se piden — a veces la IA se desvía del brief. **Adaptarse** es la regla.
2. ❌ **No usar** imágenes de baja resolución o con marca de agua.
3. ❌ **No saltarse** el paso de dar prompts — el usuario los necesita aunque él mismo vaya a generar.
4. ❌ **No inventar** URLs de imágenes que no existen.
5. ❌ **No usar** assets con licencias restrictivas en proyectos educativos o abiertos.

## 🎵 Política de audio — Música (usuario) + Sonidos (skill `sound-fx-bank`)

> **REGLA FUNDAMENTAL:** ZCode distingue claramente entre **MÚSICA** y **SONIDOS**.
>
> - 🎼 **MÚSICA** de fondo (loops ambientales, bandas sonoras) → **la provee el usuario** con IA (Suno, Udio, Stable Audio). ZCode NO genera música porque hacerlo bien requiere prompts especializados que el usuario debe dar.
> - 🔊 **SONIDOS** (efectos cortos: clicks, dings, whoosh, errores, fanfarrias, ambientes) → **ZCode los provee** vía la skill `sound-fx-bank` desde bancos libres pre-curados, con panel UI de preview y fallback sintetizado si la red falla.
> - 🎤 **VOCES** (narración TTS) → **lo provee el usuario** con ElevenLabs/PlayHT/Tortoise (igual que la música).

Si el proyecto necesita **música de fondo, efectos de sonido (SFX) o voces en off**, ZCode debe:

1. ✅ **Preguntar primero** al usuario si quiere audio (muchos usuarios prefieren silencio por defecto, accesibilidad, entornos silenciosos).
2. ✅ Ofrecer **3 caminos** (misma lógica que los assets visuales).
3. ✅ Dar **prompts optimizados** listos para copiar/pegar en generadores de audio con IA.
4. ✅ Integrar el audio con **controles visibles** (botón mute/unmute, volumen).
5. ✅ Respetar **`prefers-reduced-motion`** Y **`prefers-reduced-sound`** (autoplay bloqueado por muchos navegadores).

### Pregunta de descubrimiento (antes de implementar)

Antes de añadir audio, ZCode debe preguntar:

> 🎵 **¿Quieres añadir audio a este proyecto?**
>
> 1. **No, déjalo silencioso** (opción por defecto — accesibilidad).
> 2. **Sí, música de fondo ambiental** (loop suave durante todo el juego).
> 3. **Sí, efectos de sonido** (SFX en eventos: clicks, bingos, victoria).
> 4. **Sí, música + SFX** (experiencia inmersiva completa).
> 5. **Sí, voces en off** (narración de tarjetas, instrucciones).
> 6. **El usuario aporta archivos propios** (mp3/wav ya grabados).
>
> Para las opciones 2-5, ¿quieres que ZCode genere los prompts listos para ti?

### Opción 1 — Generar en código (preferido para SFX simples)

> **Regla de oro:** Para efectos cortos (clicks, whoosh, ding, pop), usa la **Web Audio API** del navegador — sin archivos externos.

```js
// Helper: beep / click / ding con oscilador
function sfx(name){
  if(!audioOn)return;
  const ctx=new (window.AudioContext||window.webkitAudioContext)();
  const osc=ctx.createOscillator();
  const gain=ctx.createGain();
  osc.connect(gain).connect(ctx.destination);
  const presets={
    click:{freq:1200,dur:.05,type:'square',vol:.05},
    ding:{freq:880,dur:.4,type:'sine',vol:.15,slide:440},
    whoosh:{freq:300,dur:.3,type:'sawtooth',vol:.1,slide:1200},
    pop:{freq:600,dur:.08,type:'triangle',vol:.1},
    error:{freq:200,dur:.3,type:'square',vol:.1,slide:100}
  };
  const p=presets[name]||presets.click;
  osc.type=p.type;
  osc.frequency.setValueAtTime(p.freq,ctx.currentTime);
  if(p.slide)osc.frequency.exponentialRampToValueAtTime(p.slide,ctx.currentTime+p.dur);
  gain.gain.setValueAtTime(0,ctx.currentTime);
  gain.gain.linearRampToValueAtTime(p.vol,ctx.currentTime+.01);
  gain.gain.exponentialRampToValueAtTime(.001,ctx.currentTime+p.dur);
  osc.start();osc.stop(ctx.currentTime+p.dur);
}
// Uso: sfx('ding') cuando un equipo hace bingo.
```

**Ventajas:** 0 KB de peso, latencia cero, totalmente personalizable, sin licencias.

```javascript
const SFX_CATALOG=[
  {id:'click',name:'Click suave',desc:'Botón / celda',tag:'UI',
   url:'https://assets.mixkit.co/active_storage/sfx/2571/2571-preview.mp3',
   fallback:'click'}, // mismo nombre en el preset sintetizado
  // ... 9 más
];

// 2. Precargar como elementos <audio>
const sfxElements={};
function preloadSFXBank(){
  SFX_CATALOG.forEach(s=>{
    const a=new Audio();
    a.preload='auto';
    a.volume=0.7;
    a.src=s.url;
    a.addEventListener('canplaythrough',()=>sfxElements[s.id]=a,{once:true});
    a.addEventListener('error',()=>sfxElements[s.id]=null,{once:true}); // fallback
  });
}

// 3. Reproducir: archivo real si cargó, sintetizado si no
function sfx(id){
  if(!audioOn)return;
  const el=sfxElements[id];
  if(el){el.currentTime=0;el.play().catch(()=>{});return;}
  sfxSynth(id); // fallback
}
```

#### Cómo añadir un nuevo SFX al catálogo

1. **Busca el sonido** en uno de los bancos integrados (ej: freesound.org → keyword "bell").
2. **Verifica la licencia**: prefiere CC0 o CC-BY (no CC-BY-NC para uso comercial).
3. **Copia la URL directa** del MP3/OGG.
4. **Añade entrada** al `SFX_CATALOG` con un `fallback` (nombre del preset sintetizado).
5. **Asigna un tag** para categorizar visualmente.
6. **Llama `sfx('tu-id')`** en el evento del juego correspondiente.

### 🧪 Tests que ZCode debe correr al integrar audio

1. ✅ Verificar que los archivos existen en `./audio/`.
2. ✅ Verificar peso total < 5 MB (idealmente < 2 MB con OGG).
3. ✅ Probar **mute/unmute** funciona.
4. ✅ Probar **volumen slider** funciona (rango 0-100%).
5. ✅ Verificar **autoplay policy**: el audio NO debe sonar antes de la primera interacción del usuario.
6. ✅ Verificar **preferencias del sistema** (`prefers-reduced-sound` o `prefers-reduced-motion` → audio off por defecto).
7. ✅ Probar que **SFX no se cortan** entre eventos (usar `currentTime=0;play()` no múltiples instancias).
8. ✅ Verificar **loop seamless** en música de fondo (sin clicks audibles al final).
9. ✅ Probar en **Chrome, Firefox y Safari** (cada uno maneja audio diferente).
10. ✅ Verificar que el botón de mute está **siempre visible** durante toda la experiencia.

### 🚫 Errores que NO se repiten con audio

1. ❌ **No hacer autoplay sin interacción** — los navegadores lo bloquean y el usuario pensará que está roto.
2. ❌ **No usar música con copyright** en proyectos educativos o públicos.
3. ❌ **No subir volumen** de la música por encima de las voces/SFX (regla: música siempre más baja).
4. ❌ **No usar archivos > 5 MB** — mata el tiempo de carga.
5. ❌ **No olvidarse del botón de mute** — la accesibilidad es obligatoria.
6. ❌ **No usar loops con clicks audibles** en el empalme.
7. ❌ **No generar audio con IA sin verificar la licencia** — algunos tienen restricciones comerciales.

## 💾 Política de copias de seguridad (backup)

Antes de cualquier **cambio significativo en archivos existentes**, ZCode sigue esta política de rotación:

### Cuándo crear backup

| Acción | ¿Backup? |
|--------|----------|
| Editar `index.html` con cambios > 10 líneas | ✅ Sí |
| Reescribir un archivo completo | ✅ Sí (siempre) |
| Crear archivos nuevos | ❌ No (no hay versión previa que respaldar) |
| Cambios triviales (typos, comentarios) | ❌ No |
| Cambios de configuración | ✅ Sí (si afecta producción) |

### Convención de nombres

| Tipo | Formato | Ejemplo |
|------|---------|---------|
| Backup manual del usuario | `archivo.backup` | `index.html.backup` |
| Snapshot pre-cambio | `archivo.before.{desc}_{YYYYMMDD_HHMMSS}.ext` | `index.html.before.corona_20260625_140530.html` |
| Snapshot post-bloque | `archivo.backup.bloque{N}_{YYYYMMDD_HHMMSS}.ext` | `index.html.backup.bloque3_20260625_161200.html` |

### Procedimiento estándar (Git Bash en Windows)

```bash
# 1. Snapshot del estado actual ANTES de modificar
cp archivo "archivo.before.mi_cambio_$(date +%Y%m%d_%H%M%S).ext"

# 2. Hacer las modificaciones

# 3. Si todo salió bien → snapshot del estado nuevo
cp archivo "archivo.backup.bloqueX_$(date +%Y%m%d_%H%M%S).ext"

# 4. Si algo salió mal → restaurar
cp "archivo.before.mi_cambio_YYYYMMDD_HHMMSS.ext" archivo
```

### Política de retención

- **`backup.bloque{N}_*`** → se conservan **siempre** (son hitos del proyecto).
- **`before.*`** → se pueden borrar una vez confirmado que el bloque siguiente está estable.
- **`archivo.backup`** (manual del usuario) → se conserva siempre como red de seguridad final.
- **Mantener solo las últimas 3-5 copias** de cada tipo para evitar acumulación innecesaria.

### Documentación obligatoria

Cada vez que se crea un backup de milestone (`bloque{N}_*`), actualizar la sección de historial en `MEMORIA_PROYECTO.md` con:

- Descripción breve del cambio.
- Fecha y hora.
- Tamaño del archivo antes/después.
- Razón del cambio.

### 🚫 Errores que NO se repiten con backups

1. ❌ **No sobrescribir** un archivo sin entender primero qué contiene — leer antes de escribir.
2. ❌ **No dejar** backups con el mismo nombre (siempre timestamp).
3. ❌ **No acumular** más de 10 backups del mismo tipo — limpiar los antiguos.
4. ❌ **No confiar** solo en `git` si el proyecto no es un repo git — usar snapshots manuales.
5. ❌ **No olvidar** documentar el backup en la memoria del proyecto.
